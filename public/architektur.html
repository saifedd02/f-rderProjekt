Was macht der Förderprogramm-Finder?

Ein KI-gestützter Chatbot, der deutsche Unternehmen mit passenden Förderprogrammen zusammenbringt.

1. Profil eingeben

Branche, Region, Größe, Vorhaben

→

2. Frage stellen

Nutzer schreibt im Chat was er sucht

→

3. KI-Suche

Perplexity + Gemini + lokale DB durchsuchen

→

4. Scoring

Jedes Programm bewertet (0–100 Punkte)

→

5. Ergebnis

Top 8 als verifizierte Karten

Chat-Interface

Nutzer interagieren natürlich per Chat. Mehrere Sitzungen werden gespeichert. Follow-up-Fragen werden erkannt und diversere Ergebnisse geliefert.

Hybride KI-Suche

Kombination aus lokaler Datenbank (~100 Programme) und Echtzeit-Websuche via Perplexity API. Gemini dient als Fallback und JSON-Formatierer.

Intelligentes Scoring

6 Dimensionen (Region, Größe, Branche, Bereich, Art, Text) ergeben einen Score von 0–100. Fuzzy-Matching für Region und Förderbereich. Web-Programme ab 10 Punkten, DB-Programme ab 20.

Link-Verifikation

30+ vertrauenswürdige Domains (KfW, BAFA, BMWK, Landesbanken). Jede URL wird per HEAD-Request geprüft. Warnungen bei nicht erreichbaren Links.

Lokale Speicherung

Profil, Chat-Verläufe und Favoriten werden im localStorage des Browsers gespeichert. Kein Account nötig, kein Server-Speicher.

Excel-Export

Favorisierte Programme können als .xlsx-Datei exportiert werden — bereit für Meetings und Präsentationen.

Such-Pipeline: Schritt für Schritt

Was passiert, wenn der Nutzer eine Frage stellt?

1

Prompt zusammenbauen

Nur die suchrelevanten Daten (Region, Branche, Vorhaben, Größe) werden in den Prompt gepackt — kein Firmenname, keine Mitarbeiterzahl, kein Umsatz. Bei gesetztem Region-Filter wird eine strikte Region-Einschränkung eingefügt, damit nur Programme aus dem gewählten Bundesland + bundesweite Programme zurückkommen.

Datei: /api/chat/route.ts — buildSearchPrompt()

2

Unzufriedenheit erkennen

Pattern-Matching erkennt Ausdrücke wie "andere Programme", "das passt nicht", "mehr Ergebnisse". Bei Unzufriedenheit wird die Temperatur auf 0.7 erhöht für diversere Ergebnisse und bereits gezeigte Programme ausgeschlossen.

3

KI-Suche: Perplexity (primär)

Die Anfrage geht an die Perplexity API (Modell: sonar-pro) mit aktivierter Websuche. Die Antwort enthält Fließtext + Quellen-URLs (Citations).

Datei: /lib/perplexity.ts

4

Fallback: Gemini + JSON-Formatierung

Falls Perplexity fehlschlägt, wird Gemini 2.5 Flash mit Google Search Grounding benutzt. In beiden Fällen konvertiert Gemini die Rohdaten in ein strukturiertes JSON-Schema mit Name, Beschreibung, Förderbereich, Link etc.

Datei: /lib/gemini.ts

5

Link-Verifikation

Jede URL wird gegen eine Whitelist von 30+ vertrauenswürdigen Domains geprüft (KfW, BAFA, BMWK, Landesbanken, EU-Portale). Zusätzlich wird ein HEAD-Request (4s Timeout) ausgeführt, um die Erreichbarkeit zu verifizieren.

Funktion: mapAndVerifyPrograms()

6

Scoring und Ranking

Jedes Programm wird mit dem Nutzerprofil abgeglichen und erhält einen Score von 0–100. Programme unter 10 Punkten (Web) bzw. 20 Punkten (Datenbank) werden gefiltert. Region-Matching ist fuzzy (NRW = Nordrhein-Westfalen). Die Top 8 nach Score sortiert zurückgegeben. Duplikate werden gemergt.

Datei: /lib/search.ts — scoreProgramList()

Quellen-Pipeline: Woher kommen die Programme und URLs?

Der komplette Weg eines Förderprogramms — von der KI-Suche bis zur Anzeige mit verifizierten Quellen-Links.

Phase 1 — Suchanfrage an die KI-APIs

Das System hat zwei Suchstrategien. Perplexity ist der Primärweg, Gemini der Fallback. Es wird nie beides gleichzeitig benutzt.

Weg A: Perplexity API

Modell: sonar-pro
Parameter: web_search: true

Das System schickt zwei Nachrichten an Perplexity:

1. System-Nachricht: Instruiert Perplexity als Recherche-Experten. Muss mindestens 5 Programme finden. Durchsucht foerderdatenbank.de, KfW, BAFA, BMWK, Landesförderbanken, ZIM, go-digital etc. Darf NIEMALS "nichts gefunden" sagen.

2. User-Nachricht: Fokussierter Prompt mit Suchanfrage, strikter Region-Einschränkung und minimalem Kontext (nur Branche, Größe, Vorhaben — kein Firmenname/Umsatz/Mitarbeiter).

Was kommt zurück:

// Perplexity Response { "text": "Hier sind passende Programme: 1. KfW-Digitalisierungskredit...", "citations": [ "https://www.kfw.de/inlandsf...", "https://www.foerderdatenbank...", "https://www.bmwk.de/..." ] }

Die citations sind echte URLs, die Perplexity beim Durchsuchen des Internets gefunden und als Quelle verwendet hat. Diese werden später zur Link-Verifikation genutzt.

ODER

Weg B: Gemini (Fallback)

Modell: gemini-2.5-flash
Tool: google_search

Gemini arbeitet in zwei Durchgängen:

Durchgang 1 — Grounded Search: Gemini bekommt Zugang zum Google-Search-Tool und sucht aktiv im Web. Die Antwort kommt als Fließtext (kein JSON, weil Gemini im Grounding-Modus kein zuverlässiges JSON liefert).

Die URLs stecken in den Grounding-Metadaten:

// Gemini Response Metadata candidates[0] .groundingMetadata .groundingChunks[]: [ { web: { title: "KfW-Förderung", uri: "https://www.kfw.de/..." }}, { web: { title: "BAFA-Zuschuss", uri: "https://www.bafa.de/..." }} ]

Durchgang 2 — JSON-Formatierung: Der Fließtext + die extrahierten URLs werden an Gemini geschickt (diesmal ohne Websuche) mit striktem JSON-Schema. Gemini extrahiert die Programme und gibt strukturiertes JSON zurück.

Phase 2 — Fließtext wird zu strukturiertem JSON

Egal welcher Weg (A oder B) — am Ende muss der Fließtext in ein festes JSON-Schema umgewandelt werden. Das macht die Funktion formatToJson() in gemini.ts.

Was in den Formatter reingeht:

// Input für formatToJson() 1. Fließtext von Perplexity/Gemini (bereinigt) 2. Liste der Quellen-URLs (citations / grounding URLs) 3. JSON-Schema das die Ausgabe definiert

Die entscheidende Anweisung an Gemini:

Der Formatter bekommt die Quellen-URLs als "VERIFIZIERTE QUELLEN-URLS" und die klare Regel: "Link = EXAKT aus den Quellen übernehmen ODER leer lassen. Niemals erfinden."

Was rauskommt:

// formatToJson() Output { "reply": "Ich habe 5 Programme gefunden...", "programs": [ { "name": "KfW-Digitalisierungskredit", "beschreibung": "Zinsgünstiger Kredit für...", "foerderhoehe": "Bis 25.000 EUR", "region": "Bundesweit", "link": "https://www.kfw.de/inlandsf...", "quelle": "KfW", "foerderbereich": "Digitalisierung", "foerderart": "Kredit / Darlehen", "frist": "2026-12-31", ... } ] }

Sicherheitsmechanismus: Temperatur = 0 (komplett deterministisch) + striktes JSON-Schema erzwungen via responseMimeType: "application/json". Das verhindert, dass Gemini kreativ wird.

Phase 3 — Link-Verifikation: mapAndVerifyPrograms()

Jedes Programm aus Phase 2 durchläuft jetzt eine dreistufige Verifikation. Kein Link wird ungeprüft angezeigt.

1

Domain-Whitelist

validateLink() prüft ob die Domain in der Liste der 30+ vertrauenswürdigen Domains steht. Unbekannte Domains werden sofort verworfen.

2

Citation-Matching

Falls der direkte Link fehlt oder ungültig ist, sucht matchCitationToProgram() in den Quellen-URLs nach einem passenden Link. Es vergleicht Namens-Tokens des Programms mit den URLs.

3

HEAD-Request

verifyUrl() macht einen echten HTTP-Aufruf (HEAD, 4s Timeout). Nur wenn der Server mit Status 200-399 antwortet, gilt der Link als erreichbar. Alle Links werden parallel geprüft.

Citation-Matching im Detail

Wie findet das System den richtigen Link für ein Programm?

// Beispiel: Programm "KfW-Digitalisierungskredit" 1. Name wird normalisiert und in Tokens zerlegt: "kfw", "digitalisierungskredit" 2. Jede Citation-URL wird geprüft: "https://www.kfw.de/inlandsfoerderung/Digital..." → URL enthält "kfw"? JA → validateLink() = Domain kfw.de ist trusted? JA → MATCH — diese URL wird dem Programm zugewiesen 3. Alternativ: Auch die Quelle ("KfW") wird gegen URLs geprüft

Was passiert wenn kein Link gefunden wird?

Das Programm wird trotzdem angezeigt, aber ohne Link und mit einer Link-Warnung (orangefarbener Hinweis in der ProgramCard). Der Nutzer sieht dann: "Link konnte nicht verifiziert werden".

Phase 4 — Quellen-URLs für die Anzeige (sourceUrls)

Neben dem direkten Programm-Link gibt es die sourceUrls — das sind die Quellen, die die KI bei der Suche verwendet hat. Diese werden dem Nutzer separat angezeigt.

Woher kommen die sourceUrls?

  • Bei Perplexity: Die citations aus der API-Antwort — echte URLs die Perplexity beim Websuchen als Quellen benutzt hat
  • Bei Gemini: Die groundingChunks[].web.uri aus den Grounding-Metadaten — URLs die Google Search während der Antwortgenerierung gefunden hat
  • Bei lokaler DB: Keine sourceUrls — die Programme kommen aus der eigenen Datenbank

Wie werden sie angezeigt?

In der ProgramCard.tsx werden maximal 3 sourceUrls in einem aufklappbaren "Quellen"-Bereich angezeigt. Jede URL ist ein klickbarer Link der in einem neuen Tab öffnet.

// ProgramCard.tsx — Quellen-Anzeige if (sourceUrls.length > 0) { zeige Abschnitt "Quellen" for each url in sourceUrls.slice(0, 3) → Klickbarer Link mit target="_blank" }

Phase 5 — Zusammenbau: Vom Rohdatum zur Programm-Karte

Jedes Programm bekommt am Ende eine stabile ID und alle Felder werden bereinigt.

// Finales ScoredProgram-Objekt { program: { id: "web-kfw-digitalisierungskredit-kfw", // slugified name: "KfW-Digitalisierungskredit", link: "https://www.kfw.de/...", // verifiziert quelle: "KfW", ... }, score: 87, // 0-100 source: "websuche", // oder "datenbank" / "hybrid" confidence: "medium", // web = medium, lokal = high sourceUrls: ["https://www.kfw.de/...", ...], // Quellen reasons: [ // Match-Gründe { label: "Region: Bundesweit", matched: true }, { label: "Branche: Industrie", matched: true }, ... ], deadlineStatus: "active", linkWarning: null // oder Warnungstext }

Zusammenfassung: Der komplette Weg

Nutzer-Frage ↓ buildSearchPrompt() → Profil + Filter + Verlauf einbetten ↓ searchFoerderprogramme() → Perplexity API (web_search: true) ↓ → Antwort: Fließtext + citations[] ↓ → Fallback: Gemini grounded search ↓ formatToJson() → Gemini wandelt Fließtext in JSON um ↓ → Nur Quellen-URLs als Links erlaubt ↓ → Schema erzwungen, Temperatur = 0 ↓ mapAndVerifyPrograms() ↓ validateLink() → Domain in Whitelist? ↓ matchCitationToProgram() → URL aus Citations zuordnen ↓ verifyUrl() → HEAD-Request, 4s Timeout ↓ scoreProgramList() → 6 Dimensionen bewerten (max 100) ↓ → sourceUrls anhängen ↓ → Filter: Score >= 20 ↓ mergeScoredPrograms() → Web + Lokal zusammenführen ↓ → Duplikate erkennen (Name+Quelle+Region) ↓ → Höherer Score gewinnt ↓ Top 8 sortiert nach Score → ProgramCard.tsx im Frontend

Scoring-Algorithmus

Wie wird der Match-Score für jedes Programm berechnet? Maximal 100 Punkte, verteilt auf 6 Dimensionen.

Region25 Pkt

Exakter Match = 25, "Bundesweit" = 20

Unternehmensgröße20 Pkt

Muss übereinstimmen, wenn angegeben

Branche20 Pkt

Exakt = 20, "Alle Branchen" = 16

Förderbereich15 Pkt

Normalisierter Textvergleich

Förderart10 Pkt

Zuschuss, Kredit, Coaching, etc.

Text-Ähnlichkeit25 Pkt

Keyword-Overlap aus Anfrage + Vorhaben

Match-Qualitätsstufen

Basierend auf dem Score werden Programme in Stufen eingeteilt:

85%+ Top-Empfehlung 65%+ Gut passend 40%+ Bedingt passend <40% Wenig relevant

Normalisierungs-Pipeline

Vor dem Scoring werden alle Texte normalisiert:

  • Lowercase + Akzent-Entfernung (NFD)
  • Stopwort-Filter (deutsche Artikel, etc.)
  • Alias-Auflösung ("Darlehen" → "Kredit / Darlehen")

Frist-Erkennung

Unterstützte Datumsformate:

  • ISO: 2026-12-31
  • Deutsch: 31.12.2026
  • Text: "laufend", "fortlaufend"
  • Status: aktiv, bald ablaufend (≤60 Tage), abgelaufen

Merge-Logik

Programme aus verschiedenen Quellen (DB + Web) werden zusammengeführt:

  • Gruppierung nach normalisiertem Schlüssel
  • Höherer Score gewinnt
  • Gründe und Links werden gemergt
  • Quelle wird "hybrid"

Datenquellen und Vertrauen

Die drei Quellen für Förderprogramme und wie sie kombiniert werden.

● Lokale Datenbank

~100+ Programme, manuell kuratiert in foerderprogramme.ts. Enthält KfW, BAFA, Landesbanken, EU-Programme mit vollständigen Details.

Vertrauen: HOCH

Vollständige Felder: Name, Beschreibung, Förderhöhe, Zielgruppe, Region, Frist, Förderbereich, Branche, Link

● Perplexity Websuche

Echtzeit-Suche im Internet nach aktuellen Förderprogrammen. Liefert Quellen-URLs (Citations) für Verifikation. Modell: sonar-pro (beste Qualität). Prompt enthält strikte Region-Einschränkung und minimalen Kontext.

Vertrauen: MITTEL

Nur Programme mit verifizierten Links auf vertrauenswürdige Domains werden als "mittel" eingestuft

● Gemini (Fallback)

Google Gemini 2.5 Flash mit Search Grounding. Wird aktiviert wenn Perplexity fehlschlägt. Dient auch als JSON-Formatierer für Perplexity-Ergebnisse.

Vertrauen: MITTEL

Dual-Rolle: Eigenständige Suche (Fallback) + Strukturierte Formatierung der Perplexity-Rohdaten

Merge-Strategie: Hybrid-Programme

Wenn ein Programm aus der lokalen DB und der Websuche gefunden wird:

  • Gruppierung nach normalisiertem Schlüssel (Name + Quelle + Region)
  • Die Version mit dem höheren Score gewinnt
  • Match-Gründe und Links werden zusammengeführt
  • Quelle wird als "hybrid" markiert
